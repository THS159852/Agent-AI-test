# Test Case Reviewer

Review **existing** test cases from `testcase-generator`. Do **not** rewrite the full suite unless the user asks for fixes. Follow `docs/QA_GLOBAL_RULES.md` and the rules in `skills/testcase-generator/SKILL.md`.

## Required inputs

| Input | Required | Source |
|-------|----------|--------|
| Generated test cases | **Yes** | Chat output, `@file`, or pasted table |
| Original requirements / AC / BR | **Yes** (preferred) | Same sources used to generate |
| UI / browser document | When UI cases exist | Mockups, `browser-test-document.md`, screenshots |
| Coverage summary from generator | Optional | Helps spot claimed-vs-actual gaps |

If requirements are missing, review **format/quality only**, mark source-alignment findings as `Blocked — missing source`, and ask for the source files.

## Workflow

```
Progress:
- [ ] Step 1: Inventory screens and map every TC
- [ ] Step 2: Quality gate (format / language / structure)
- [ ] Step 3: Source alignment (requirement match)
- [ ] Step 4: Coverage & mismatch analysis (per screen)
- [ ] Step 5: Write review report (grouped by screen)
```

### Step 1: Inventory screens and map every TC

Build a **Screen Registry** before scoring findings:

| Screen ID | Screen name | URL / route (if known) | Module column values | TC IDs on this screen |
|-----------|-------------|------------------------|----------------------|------------------------|

**How to resolve screen:**

1. Prefer `Module` column when it names a screen/feature.
2. Else derive from Steps (`Navigate to /login` → Login screen).
3. Else from Requirement Ref / browser doc `SCR-###`.
4. Else from Title subject (`Verify login…` → Login).
5. If still unclear → Screen = `Unmapped` and raise a finding.

Every finding **must** include **Screen**.

### Step 2: Quality gate

Check each test case against generator + global rules:

| Check | Fail when |
|-------|-----------|
| Language | Any column not English (esp. Expected Result) |
| Title | Missing `Verify` prefix, or chains unrelated checks with "and" |
| ID | Missing / duplicate / wrong convention |
| Requirement Ref | Empty and not marked `[Derived]` |
| Steps | Vague ("perform login"), not numbered, missing data |
| Expected Result | Vague ("works correctly", "OK", "as expected") |
| Test Data | Placeholder-only (`valid email`) instead of concrete values |
| Priority / Type / Auto Candidate | Missing or inconsistent with risk |
| One verification | One TC asserts multiple unrelated outcomes |

### Step 3: Source alignment

For each TC, compare Steps + Expected Result to the cited AC/BR/UI:

| Issue type | Meaning |
|------------|---------|
| **Mismatch** | Expected result or steps contradict the requirement / observed UI |
| **Over-spec** | Invents rules not in source (e.g. max length 50 when unspecified) |
| **Under-spec** | Requirement exists but Expected Result is too weak to verify it |
| **Wrong screen** | TC content belongs to another screen than Module/Steps claim |
| **Stale / N/A** | Tests out-of-scope or removed behavior |
| **Duplicate** | Same condition covered by another TC |

Never invent missing business rules — list them under Questions.

### Step 4: Coverage & mismatch analysis (per screen)

For **each screen**, evaluate:

| Dimension | Look for |
|-----------|----------|
| AC coverage | AC on this screen with no TC |
| Positive / Negative / Boundary | Missing for validated fields |
| Permission / Role | Missing when roles are in scope |
| UI vs API mix-up | UI expected results on API-only AC (or reverse) |
| Traceability | TC cites wrong AC or no AC |
| Priority skew | Critical happy path as P2/P3, or trivial as P0 |

### Step 5: Write review report

Report language: match the user (Vietnamese if they asked in Vietnamese). Keep **TC IDs and Titles in English** as authored.

Use this template:

```markdown
## Test Case Review Summary
- Sources reviewed: <files / docs>
- Total test cases: N
- Screens covered: M
- Findings: Critical C / Major Ma / Minor Mi
- Verdict: Pass | Pass with fixes | Rework required

## Screen Registry
| Screen ID | Screen name | TC count | Finding count |
|-----------|-------------|----------|---------------|

## Findings (by screen)

### Screen: <Screen name> (<Screen ID>)
| Finding ID | Severity | TC ID | Title | Issue type | Problem | Evidence (AC/UI) | Suggested fix |
|------------|----------|-------|-------|------------|---------|------------------|---------------|
| REV-001 | Critical | LOGIN-TC-003 | Verify … | Mismatch | … | AC-2 | … |

### Screen: Unmapped
| … |

## Coverage gaps (by screen)
| Screen | Missing condition | Related AC/BR | Suggested new TC title |
|--------|-------------------|---------------|------------------------|

## Quality defects (cross-screen)
| Finding ID | Severity | TC ID | Screen | Problem | Suggested fix |
|------------|----------|-------|--------|---------|---------------|

## Duplicates
| TC A | TC B | Screen | Same condition |
|------|------|--------|----------------|

## Assumptions / Questions / Risks

## Suggested next steps
1. Fix Critical/Major findings in place, or re-run `testcase-generator` for affected screens only
2. Re-run this skill on the updated pack
3. Optional: `testdata-generator` / `automation-script-writer` after Pass
```

### Severity

| Severity | Use when |
|----------|----------|
| **Critical** | Wrong expected result vs AC; security/permission hole; wrong screen that would fail execution |
| **Major** | Missing P0 coverage; invented rule; unusable steps; non-English Expected Result |
| **Minor** | Wording, ID style, Auto Candidate, non-blocking duplicate nuance |

### Verdict rules

| Verdict | When |
|---------|------|
| **Pass** | No Critical/Major; Minor optional |
| **Pass with fixes** | Only Minor, or Major that is quick to patch |
| **Rework required** | Any Critical, or Major covering ≥1 full screen / AC cluster |

## Anti-patterns

| Avoid | Do instead |
|-------|------------|
| Regenerating all TCs by default | Report findings; regenerate only if user asks |
| Findings without Screen | Always set Screen / `Unmapped` |
| Vague "testcase is wrong" | Cite TC ID + AC/UI evidence + suggested fix |
| Silent invented rules during review | Flag as Over-spec or ask Questions |
| Ignoring Module vs Steps conflict | Raise **Wrong screen** |
